We believe we have all the hardware figured out for the positioning system. I’m going to copy and paste the description I sent to Mr. Williams:

There are 5 Arduino microcontrollers (Arduino Uno). Each microcontroller has a radio module. The radio module can send and receive radio signals with 300 different channels (or something like that). There will be 4 microcontrollers set up around the court perimeter, and these are the "anchors" (as an end product we will want to triangulate, but for a prototype we want to keep it simple -- a square is much easier to compute position). The user has the 5th microcontroller. The users radio module emits a pulse, and the anchors receive the pulse. All the microcontrollers have clocks, so depending on the time it takes to receive the signal we can compute a distance perpendicular to each anchor (radio signals travel at the speed of light d = r*t). Subsequently the 4 anchors will communicate with a raspberry Pi (a small UNIX microcontroller -- we already have some of these), and the raspberry pi will compute position. The raspberry pi will serve as the "central hub" for all embedded systems.

There are also other various parts -- batteries, power supplies, and voltage droppers that will be needed.

These are all the features we are thinking about for the mobile application.

One thing that we haven’t figured out is how to determine who’s shot when in given there is more than 1 person playing. Since there is so much to this project, we are going to plan on having a prototype for the end result. That prototype will consist of a positioning system, a backboard speaker (for direction), and a system to tell the user where the shot landed. All of the above will be for a single user.

Sunday Oct. 2nd 2016

Met with the team today and got a lot of good things figured out. We are all now on the same page for the functional requirements of the hardware system. We are also on the same page about the milestones for the smartphone applications. I want to clarify exactly what our end prototype should consist of.



Hardware system

  • Positioning system
    • User's microcontroller emits signal
    • Each anchor microcontroller receives signal
    • Each anchor microcontroller calculates distance
    • Each anchor microcontroller sends distance packet to central processing unit
    • Central processing unit computes positioning
    • Central processing unit sends bluetooth packet to smartphone
  • Distance system
    • Central processing unit collects and computes position
    • Position of goal is constant
    • Central processing unit subtracts unit coordinates from goal coordinates
    • Central processing unit sends distance measurement to smartphone
  • Shot made? system
    • Microcontroller on backboard has IR sensor
    • If IR sensor activated: shot made
    • Then send packet to central processing unit
    • Central processing unit sends shot made packet to smartphone


Milestones for smartphone applications

  • User authentication
  • Statistics
  • Games -- HORSE


Note on the smartphone applications

After doing some research, we have decided to utilize the Firebase service. Firebase is a service that integrates with Google Cloud Platform and handles all backend services. This will make things WAY simpler. Simpler is good for several reasons: quicker development (we are busy students) and easier for the high school students to follow.

After the little bit of iOS dev I did over the weekend, I'm excited to get started. Midterm weeks are coming up these next two weeks, so development might slow down a little. After midterms are finished, I should have time to crank out quite a bit of development.
